## 1. 사용 API 범위

### 1-A. 네이버 지역 검색 API

> 공식 문서: https://developers.naver.com/docs/search/local/

| 항목 | 내용 |
|---|---|
| 목적 | 카페명·주소 실존 확인 및 후보 발굴 |
| 요청 파라미터 | `query`, `display` (최대 5), `start` (최대 1~1000) |
| 저장 가능 필드 | `title` (카페명 후보), `address` / `roadAddress`, `category`, `telephone` |
| 저장 금지 필드 | `description` (리뷰 발췌), `link` (네이버 플레이스 URL — 참조만 가능) |
| 용도 | 후보 카페 실존 확인, 주소 정규화 기준 |

### 1-B. 네이버 블로그 검색 API

> 공식 문서: https://developers.naver.com/docs/search/blog/

| 항목 | 내용 |
|---|---|
| 목적 | 카공 언급 게시물에서 카페명·속성 키워드 추출 |
| 요청 파라미터 | `query`, `display` (최대 100), `sort` (sim/date) |
| 저장 가능 필드 | `title` (게시물 제목), `description` (요약 발췌, 최대 1문장 — 속성 추출 후 폐기), `bloggername` (참고용), `postdate` |
| 저장 금지 필드 | 게시물 본문 전체, 이미지 URL, 작성자 식별 정보 |
| 속성 추출 방식 | `title` + `description` 텍스트에서 키워드 매칭 → 속성 키만 저장 (원문 폐기) |

### 1-C. 네이버 카페글 검색 API

> 공식 문서: https://developers.naver.com/docs/search/cafearticle/

| 항목 | 내용 |
|---|---|
| 목적 | 지역 커뮤니티에서 카공 카페 추천 게시물 탐색 |
| 요청 파라미터 | `query`, `display` (최대 100), `sort` (sim/date) |
| 저장 가능 필드 | `title`, `description` (요약 발췌, 속성 추출 후 폐기), `cafename`, `postdate` |
| 저장 금지 필드 | 게시물 본문 전체, 이미지, 카페(community) 회원 정보 |

## 3. 검색 키워드 생성 방식

키워드는 `src/utils/candidateKeywords.ts`에서 관리합니다.

### 기본 키워드 패턴

```
{지역} + {카공 의도 표현}
```

| 범주 | 예시 |
|---|---|
| 지역 기반 | 인천 카공, 인천 노트북 카페, 인천 공부 카페 |
| 구 기반 | 연수구 카공, 부평구 카공, 남동구 카공 |
| 랜드마크/대학 기반 | 인하대 카공, 인천대 카공, 송도 카공 |
| 속성 기반 | 인천 콘센트 카페, 인천 24시간 카페, 인천 조용한 카페 |

### 키워드 조합 규칙

1. 지역어 × 카공 의도어 조합으로 생성 (예: "검단" × "카공")
2. 너무 일반적인 키워드는 제외 (예: "카페" 단독 — 결과 무관 후보 과다)
3. 중복 제거 후 고유 키워드 배열로 반환

## 5. 후보명/주소 정규화 규칙

### 카페명 정규화

1. HTML 태그 제거: `<b>`, `</b>` 등
2. 특수문자 정리: `&amp;` → `&` 등 HTML 엔티티 디코딩
3. 괄호 내 부가 정보 제거: `브루웍스 (송도점)` → `브루웍스 송도점`
4. 앞뒤 공백 제거

### 주소 정규화

1. `인천광역시` 또는 `인천`으로 시작하는지 확인 (타 지역 필터링)
2. 도로명 주소 우선, 없으면 지번 주소 사용
3. 구/동 파싱: `인천광역시 연수구 송도동` → `district: "연수구"`, `dong: "송도동"`

## 6. 중복 제거 기준

수집 파이프라인에서 동일 후보가 여러 키워드로 중복 수집될 수 있습니다.

| 단계 | 기준 |
|---|---|
| 1차 (이름 기반) | 정규화된 카페명이 동일한 경우 → 최신 항목만 유지 |
| 2차 (주소 기반) | 도로명 주소가 동일한 경우 → 병합 또는 중복 플래그 |
| 3차 (수동) | 운영자 검수 시 기존 `cafes` 테이블과 대조 |

> 현재 단계: 1차 중복 제거는 스크립트에서 처리. 2·3차는 운영자 검수 시 수행.

## 7. 카공 속성 후보 추출 방식

게시물 `title` + `description` (요약 발췌)에서 키워드 매칭으로 속성 후보를 추출합니다.

추출 결과는 **속성 키와 신뢰도만 저장**하고 원문은 즉시 폐기합니다.

```
입력:  "콘센트 있고 조용한 송도 카페 추천"
출력:  { outlet: 0.9, quiet: 0.9 }   // 속성 키 + confidence만
폐기:  "콘센트 있고 조용한 송도 카페 추천"  // 원문
```

속성 추출 로직은 `src/utils/candidateAttributeExtractor.ts`에서 관리합니다.

## 8. 운영자 검수 큐 등록 방식

### 등록 흐름

```
API 응답 수신
  → 카페명/주소 정규화
  → 중복 체크 (기존 raw_cafe_candidates와 대조)
  → 카공 속성 키워드 추출 (원문 폐기)
  → confidence_score 계산
  → raw_cafe_candidates INSERT (status: 'pending')
  → 운영자 검수 대기
```

### confidence_score 계산 기준

| 조건 | 점수 |
|---|---|
| 속성 키워드 2개 이상 추출됨 | +0.3 |
| 네이버 지역 검색에서 '카페' 카테고리로 확인됨 | +0.4 |
| 인천광역시 주소 확인됨 | +0.2 |
| 최근 6개월 내 게시물 | +0.1 |

> 합계 0.6 이상 → 고신뢰 후보 (운영자 우선 검토 권장)

## 9. API Rate Limit 고려사항

| API | 기본 한도 | 권장 처리 방식 |
|---|---|---|
| 네이버 검색 API (통합) | 25,000 calls/day | 키워드당 1회, 하루 최대 100 키워드 |
| 블로그/카페글 검색 | display 최대 100 | `display=10`으로 제한, 필요 시 페이징 |
| 지역 검색 | display 최대 5 | 카페명 검증 용도로만 사용 |

### 수집 스케줄 권장안

- **최초 1회**: 전체 키워드(~100개) 순차 수집
- **이후**: 주 1회 신규 키워드 또는 날짜 기반 최신 게시물만 수집
- **Rate limit 초과 방지**: 요청 간 300ms 이상 딜레이 삽입
