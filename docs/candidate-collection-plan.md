# 카공 카페 후보 수집 자동화 설계 문서

> 작성일: 2026-04-28  
> 이 문서는 인천 카공 카페 데이터를 수집·검수·반영하는 전체 파이프라인을 정의합니다.  
> **공식 API 기반 수집 + 운영자 수동 검수**를 전제로 하며, 비공식 크롤링 및 원문 복제는 전면 금지입니다.

---

## 1. 자동화 목적

자동화의 목적은 **외부 콘텐츠를 복제하는 것이 아닙니다.**

자동화는 다음 목적에만 사용합니다.

| 목적 | 설명 |
|------|------|
| 후보 카페 발굴 | 인천에서 카공 가능한 카페의 이름과 위치를 수집 |
| 중복 후보 정리 | 다른 소스에서 동일한 카페가 중복 수집되는 것을 방지 |
| 네이버 지도 존재 검증 | 수집된 후보가 실제 영업 중인지 확인 |
| 카공 속성 후보 추출 | 콘센트, 와이파이, 조용함 등 관련 키워드 분류 |
| 운영자 검수 큐 생성 | 사람이 직접 확인해야 할 후보 목록 생성 |

**자동화가 대체할 수 없는 것**: 카공 점수 최종 판단, 운영 정보 확인, 데이터 공개 승인

## 2. 허용 가능한 데이터 소스

### 공식 API 소스

| 소스 | API | 수집 가능 정보 | 제한 |
|------|-----|--------------|------|
| 네이버 블로그 | 네이버 검색 API (blog) | 카페명 언급, 카공 키워드 | 본문 복사 금지 |
| 네이버 카페 | 네이버 검색 API (cafe) | 카페명 언급, 카공 키워드 | 본문 복사 금지 |
| 네이버 지역 | 네이버 지역 검색 API | 상호명, 주소, 전화번호, 카테고리 | 리뷰 저장 금지 |
| 인스타그램 | Instagram Graph API (해시태그) | 게시물 위치 태그, 해시태그 | 이미지/본문 저장 금지 |

### 수동 소스

- 운영자 직접 조사 (방문, 포털 검색)
- 사용자 카페 제안 (`user_suggestions` 테이블)
- 지역 커뮤니티 제보 (검수 후 수동 등록)

### `source_type` 값 매핑

```sql
-- raw_cafe_candidates.source_type
'naver_blog'        -- 네이버 블로그 검색 API
'naver_cafe'        -- 네이버 카페 검색 API  
'naver_local'       -- 네이버 지역 검색 API
'instagram'         -- Instagram Graph API
'manual'            -- 운영자 수동 등록
'user_suggestion'   -- 사용자 제안 (user_suggestions → 전환)
```

## 3. 금지되는 수집 방식

다음은 어떤 이유로도 허용하지 않습니다.

### 절대 금지

| 금지 항목 | 이유 |
|----------|------|
| 네이버/구글/카카오 리뷰 원문 복사 | 저작권 침해, 플랫폼 이용약관 위반 |
| 외부 평점(별점) 숫자 그대로 저장 | 출처 오인 가능성, 데이터 품질 왜곡 |
| 외부 사진 무단 저장 | 저작권 침해 |
| SNS 게시물 본문/이미지 저장 | 저작권 침해, Meta 이용약관 위반 |
| 비공식 크롤러(puppeteer, selenium 등) | 서버 부하 유발, 이용약관 위반 |
| 타 앱 데이터셋 직접 이관 | 저작권/계약 위반 가능성 |
| JavaScript 렌더링 기반 스크래핑 | 비공식 크롤링에 해당 |
| 로그인 우회 수집 | 불법 접근에 해당 |

### 허용 범위 명확화

```
허용: 카페명, 주소, 전화번호, 카테고리 (공식 API 응답)
허용: 카공 관련 키워드 등장 여부 (예: "콘센트 있어요")
허용: 게시물 존재 여부, 해시태그 빈도

금지: 리뷰 본문 전문
금지: 별점/평점 숫자
금지: 사진 URL 저장
금지: 작성자 정보
```

## 4. 키워드 생성 전략

### 4.1 기본 카공 키워드

```
인천 카공
인천 카페 공부
인천 공부하기 좋은 카페
인천 노트북 카페
인천 콘센트 카페
인천 24시간 카페
인천 스터디 카페 말고 카페
```

### 4.2 지역 키워드 (구/동 + 카공)

```
송도 카공          부평 카공
구월동 카공        청라 카공
주안 카공          검단 카공
계양 카공          연수동 카공
동인천 카공        인하대 카공
인천대 카공        학익동 카공
```

### 4.3 속성 특화 키워드

```
인천 콘센트 많은 카페
인천 와이파이 빠른 카페
인천 조용한 카페 공부
인천 늦게까지 여는 카페
인천 24시 카페
```

### 4.4 키워드 조합 원칙

- 지역 키워드 × 속성 키워드 조합으로 쿼리 생성
- 구 단위 키워드 우선, 동 단위 키워드로 세분화
- 동일 쿼리 중복 실행 방지 (7일 내 재실행 금지)

## 5. 후보 정규화 방식

API 응답에서 후보 카페를 추출한 뒤 다음 기준으로 정규화합니다.

### 5.1 카페명 정규화

```
정규화 대상:
- 앞뒤 공백 제거
- 특수문자 통일 (& → &, · → ·)
- 지점명 분리 (예: "온도카페 송도점" → 이름: "온도카페", 메모: "송도점")
- 초성/로마자 병기 제거 (예: "카페온도(Cafe Ondo)" → "카페온도")
```

### 5.2 주소 정규화

```
정규화 대상:
- 도로명/지번 주소 통일 (네이버 지역 API 기준 도로명 우선)
- "인천광역시" 앞부분 통일
- 층수/호수는 별도 필드로 분리하지 않음 (단순 텍스트 보관)
```

### 5.3 저장 구조 (`raw_cafe_candidates`)

```sql
candidate_name     -- 정규화된 카페명
candidate_address  -- 정규화된 주소 (도로명 우선)
candidate_url      -- 소스 게시물 URL (본문 아님, 참고용)
source_keyword     -- 검색에 사용된 키워드
extracted_keywords -- 카공 속성 관련 키워드 배열 (예: ['콘센트', '와이파이'])
confidence_score   -- 0.0~1.0 신뢰도
```

## 6. 중복 제거 방식

### 6.1 1차: 정확 매칭 (DB)

```sql
-- 동일 카페명 + 동일 주소 조합이 이미 raw_cafe_candidates에 있으면 스킵
SELECT COUNT(*) FROM raw_cafe_candidates
WHERE candidate_name = :name AND candidate_address = :address;
```

### 6.2 2차: 유사 매칭 (정규화 후)

다음 경우를 동일 후보로 간주합니다.

| 케이스 | 예시 | 처리 |
|--------|------|------|
| 지점명 차이 | "카페온도" vs "카페온도 송도점" | 주소로 재확인 |
| 영문/한글 혼용 | "Cafe Ondo" vs "카페온도" | 별도 수동 확인 |
| 공백/특수문자 | "카 페 온 도" vs "카페온도" | 공백 제거 후 비교 |

### 6.3 3차: 네이버 지역 검색 검증

중복 후보는 네이버 지역 검색으로 플레이스 ID를 확인하여 동일 장소 여부를 판단합니다.

---

## 7. 네이버 지역 검색 검증 방식

모든 후보는 최종 DB 반영 전 네이버 지역 검색으로 검증합니다.

### 7.1 검증 목적

- 실제 영업 중인 카페인지 확인
- 정확한 주소/위도경도 확보
- 네이버 플레이스 URL 확보 (`naverMapUrl`)

### 7.2 검증 절차

```
1. 네이버 지역 검색 API 호출
   Query: "{candidate_name} {candidate_address 앞 2~3 토큰}"
   
2. 응답 결과 중 가장 유사한 항목 선택
   - 카테고리가 "카페" 또는 "커피전문점" 포함
   - 주소가 후보 주소와 일치 또는 유사

3. 검증 성공 시 저장 항목
   - 확정 카페명 (네이버 기준)
   - 확정 주소 (도로명)
   - 위도 / 경도
   - 네이버 플레이스 URL

4. 검증 실패 시
   - review_status = 'pending' 유지
   - reviewer_note = '지역 검색 결과 없음' 기록
   - 운영자가 수동으로 확인
```

### 7.3 신뢰도 점수 기준

| 조건 | confidence_score |
|------|-----------------|
| 카페명 + 주소 완전 일치 | 0.9 이상 |
| 카페명 일치, 주소 유사 | 0.7 ~ 0.89 |
| 카페명 유사, 주소 일치 | 0.5 ~ 0.69 |
| 하나만 일치 | 0.5 미만 |

## 8. 카공 속성 후보 추출 방식

리뷰 본문을 저장하지 않고, **키워드 등장 여부**만 추출합니다.

### 8.1 추출 대상 키워드

```
콘센트 관련:   콘센트, 충전, 멀티탭, 플러그
와이파이 관련: 와이파이, wifi, 무선인터넷
조용함 관련:   조용한, 조용해요, 조용합니다, 집중
혼자 관련:     혼자, 1인석, 혼카공
시간 관련:     24시간, 늦게까지, 새벽, 야간
음료 관련:     커피, 라떼, 에스프레소, 원두
디저트 관련:   케이크, 베이글, 디저트, 빵
그룹 관련:     스터디, 팀플, 모임, 단체
```

### 8.2 추출 및 저장 방식

```
1. API 응답의 제목/스니펫(snippet) 텍스트에서 키워드 탐색
   (게시물 본문 전체 저장 금지)

2. 등장한 키워드를 extracted_keywords 배열에 저장
   예: ['콘센트', '와이파이', '조용한']

3. 키워드 수에 따라 confidence_score 가산
   - 카공 관련 키워드 3개 이상: +0.1
   - 카공 관련 키워드 1~2개: +0.05
```

### 8.3 카공 점수 최종 결정 원칙

추출된 키워드는 **힌트**일 뿐이며, 운영자가 직접 방문하거나 추가 확인 후 최종 점수를 설정합니다.  
자동 추출 키워드를 카공 점수로 직접 변환하지 않습니다.

## 9. 운영자 검수 큐 흐름

```
[자동 수집]
    │
    ▼
raw_cafe_candidates (review_status = 'pending')
    │
    ├── confidence_score >= 0.7 → 검수 우선순위 높음
    └── confidence_score <  0.7 → 검수 우선순위 낮음
    │
    ▼
[운영자 검수 화면 - Phase 2 구현 예정]
    │
    ├── 카페 실존 여부 확인 (네이버 지도 직접 확인)
    ├── 카공 가능 여부 판단
    ├── 카공 속성 점수 직접 입력 (0~5)
    ├── 태그 선택
    └── 한 줄 요약 작성
    │
    ├── 승인 → review_status = 'approved' → cafes 테이블 반영
    └── 반려 → review_status = 'rejected' + reviewer_note 기록
    │
    ▼
[cafes 테이블 INSERT]
    │
    ├── cafes (기본 정보)
    ├── cafe_attributes (점수: 운영자 직접 입력)
    └── cafe_tags (태그: 운영자 선택)
```

### 검수 우선순위 기준

| 우선순위 | 조건 |
|---------|------|
| 높음 | confidence_score >= 0.7 AND 카공 키워드 3개 이상 |
| 보통 | confidence_score >= 0.5 |
| 낮음 | confidence_score < 0.5 또는 주소 불명확 |
| 보류 | 네이버 지역 검색에서 카페를 찾지 못한 경우 |

## 10. 최종 DB 반영 기준

운영자가 후보를 승인할 때 다음 조건을 모두 충족해야 `cafes` 테이블에 반영합니다.

### 필수 조건

| 항목 | 기준 |
|------|------|
| 카페명 | 공식 상호명으로 확정 |
| 주소 | 도로명 주소 확인 완료 |
| 위도/경도 | 네이버 지역 검색으로 확보 |
| 카공 가능 여부 | 운영자 직접 판단 |
| status | `active` (영업 중) |

### 권장 조건 (없으면 pending 유지)

| 항목 | 기준 |
|------|------|
| 카공 점수 | soloScore, quietScore, outletScore 중 1개 이상 |
| 영업시간 | openHoursSummary 입력 |
| 태그 | 1개 이상 선택 |

### 반려 기준

| 사유 | 처리 |
|------|------|
| 카페가 아닌 경우 (식당, 베이커리만 등) | rejected |
| 이미 cafes에 동일 카페 존재 | rejected + 기존 카페 ID 연결 메모 |
| 폐업/이전 확인 | rejected |
| 주소 확인 불가 | pending 유지 (보류) |

## 부록 A. 파이프라인 전체 흐름

```
키워드 생성
  └── 기본 키워드 × 지역 키워드 조합

공식 API 검색
  ├── 네이버 블로그 검색 API
  ├── 네이버 카페 검색 API
  └── Instagram Graph API (해시태그)

후보 카페명 추출
  └── API 응답의 제목/스니펫에서 카페명 패턴 추출
      (본문 전문 저장 금지)

후보 정규화
  ├── 카페명 정규화
  └── 주소 정규화

1차 중복 제거
  └── raw_cafe_candidates 내 동일 name+address 스킵

네이버 지역 검색 검증
  ├── 성공: 위도/경도/공식명/플레이스 URL 확보
  └── 실패: pending 유지, 운영자 수동 확인

카공 속성 키워드 추출
  └── API 스니펫에서 키워드 등장 여부만 기록

운영자 검수 큐 등록
  └── raw_cafe_candidates.review_status = 'pending'

운영자 승인
  └── cafes + cafe_attributes + cafe_tags INSERT

앱에 반영
  └── cafeService.getCafes() → status='active' 카페 노출
```

## 부록 B. 관련 DB 테이블 참조

| 테이블 | 역할 |
|--------|------|
| `raw_cafe_candidates` | 자동 수집 후보 풀 |
| `user_suggestions` | 사용자 제안 (별도 검수 흐름) |
| `cafes` | 최종 승인된 카페 (status='active') |
