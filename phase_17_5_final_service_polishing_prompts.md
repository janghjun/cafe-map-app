# 카공 어디가? 인천편 — Phase 17.5 최종 서비스 디벨롭 프롬프트 팩

> 사용 시점: Phase 17 완료 후, Phase 18 앱인토스 제출 패키지 작성 직전  
> 목적: 앱인토스 제출 전 마지막으로 서비스 완성도를 높이기 위한 디벨롭 단계  
> 방향: “기능 많이 추가”가 아니라, 레퍼런스 서비스에서 검증된 신뢰·지도감·최신성·제보·FAQ 구조를 우리 서비스에 맞게 최소 구현합니다.

---

## 0. 레퍼런스 인사이트 요약

### 카공맵 계열

- 핵심은 “공부/작업하기 좋은 카페”라는 목적 특화입니다.
- 카페 일반 정보보다 조용함, 콘센트, 와이파이, 좌석, 혼잡도, 체류 적합성이 중요합니다.
- 우리 서비스 반영:
  - 카공 적합도 요약 유지
  - 추천 기준 안내 강화
  - 카공 속성 중심 상세 화면 유지

### Coffice 계열

- 핵심은 콘센트, 공간 크기, 단체석, 24시간 여부, 제보, 저장입니다.
- 우리 서비스 반영:
  - 추천 카드 핵심 지표 압축
  - 상세 화면 속성 요약
  - 제보/저장 UX 강화

### 두쫀쿠맵

- 핵심은 “위치 기반 + 최신성 + 상태 정보 + 입점/제보 흐름”입니다.
- 재고 정보는 사장님이 직접 업데이트하며, 사용자는 마지막 업데이트 시간을 확인해야 한다는 FAQ 구조가 있습니다.
- 우리 서비스 반영:
  - 카페 검증 상태
  - 마지막 확인일
  - 사용자에게 정보 정확도 한계를 명확히 안내
  - 카페 제안/검수 흐름 강화

### 거지맵

- 핵심은 “명확한 목적 + 지도 기반 큐레이션 + 사용자 제보”입니다.
- 모든 식당 정보가 아니라, “가성비 식당”이라는 좁은 기준이 강점입니다.
- 우리 서비스 반영:
  - 전체 지도 플랫폼이 아니라 “카공 목적 지도/추천”
  - 추천 결과에 지도감만 보조적으로 제공
  - 사용자 제보와 운영자 검수 유지

### 네이버 지도 리스트/작업 카페 공유 사례

- 폐업/이전 때문에 정보가 쉽게 낡습니다.
- 네이버 지도 같은 장소 플랫폼은 폐업/이전 정보가 비교적 최신으로 유지될 수 있습니다.
- 우리 서비스 반영:
  - 실제 존재 여부를 완벽 자동 판단한다고 말하지 않기
  - 네이버 지도 링크/검증 출처/마지막 확인일을 통해 신뢰 레이어 구축
  - 추천 노출은 curated 또는 verified 상태 중심

# Step 17.5-2. 카페 검증 상태 모델 추가

```md
## 작업 목표
카페가 실제 존재하고 운영자가 확인한 정보인지 사용자가 알 수 있도록 카페 검증 상태 모델을 추가해 주세요.

## 현재 상태
- `Cafe` 타입과 mock 데이터가 있습니다.
- 카페 제안/관리자 검수 구조가 있습니다.
- 실제 카페 존재 여부를 완벽 자동 판단하지는 않습니다.

## 요구사항
- `src/types/cafe.ts`의 Cafe 타입에 아래 필드를 추가해 주세요.
  - `verificationStatus`
  - `lastVerifiedAt`
  - `verificationSources`
- 타입 예시는 아래를 참고하되, 기존 코드와 충돌하지 않게 조정해 주세요.

```ts
type CafeVerificationStatus =
  | "candidate"
  | "verified_basic"
  | "curated"
  | "needs_recheck"
  | "closed";

type CafeVerificationSources = {
  naverLocal?: boolean;
  kakaoLocal?: boolean;
  manualCheck?: boolean;
  userSuggestion?: boolean;
};
```

- 각 상태 의미를 주석으로 작성해 주세요.
  - candidate: 후보
  - verified_basic: 장소명/주소 수준 확인
  - curated: 운영자 검수 완료, 추천 노출 가능
  - needs_recheck: 재확인 필요
  - closed: 폐업/제외
- mock 데이터에도 필드를 추가해 주세요.
- 기본 추천 노출은 `curated` 또는 `verified_basic` 중심으로 설계하고, `closed`는 제외해야 합니다.
- 현재 추천 로직에서 `closed` 또는 `needs_recheck`를 추천에서 제외하거나 낮은 우선순위로 처리해 주세요.

## 제한사항
- 실제 외부 API 검증 구현 금지
- 검증되지 않은 데이터를 verified로 단정하지 말 것
- 추천 로직을 대규모로 변경하지 말 것
- 외부 리뷰/평점/사진 복제 금지

## 변경 예상 파일
- `src/types/cafe.ts`
- `src/data/cafes.mock.ts`
- `src/utils/recommendation.ts`
- 필요 시 `src/services/cafeService.ts`

## 테스트 방법
- TypeScript 타입 체크
- `closed` 카페가 추천 결과에 나오지 않는지 확인
- `needs_recheck` 카페가 낮은 우선순위 또는 제외 처리되는지 확인
- 기존 추천 결과가 깨지지 않는지 확인

## 출력 형식
1. 변경 요약
2. 생성/수정 파일
3. 검증 상태 모델 설명
